Version V2.1.0 · 像素科技背景增强版 · 2026-05-17

小丑牌 Web · 像素科技背景增强

Product Requirements Document — 基准：index.html（单文件，GitHub Pages 已上线）  |  在 V2.0.0 二次元改造基础上，新增 Synthwave×16-bit 像素科技背景层

1改造目标

把现有西方扑克风格的小丑牌游戏，换上 B站双色调 + 萌系角色美学，让二次元向玩家在第一眼就感受到"这是为我做的游戏"，同时保留全部游戏规则和算分逻辑不变。

改造范围：视觉主色 / 卡牌标记 / 吉祥物对话 / 弹幕飘字 / 文案萌化 / 胜利粒子 / Web Audio 音效
不碰的部分：游戏规则 / 算分公式 / 关卡进程逻辑 / 小丑牌效果数值

2目标用户与场景

维度描述
目标用户 18–28 岁、日常刷 B站/看番/玩 Galgame 的轻度玩家；对 Balatro 原版不熟悉，但被"卡牌养成"类型吸引
核心场景 下课/下班后打开浏览器，想打发 5–10 分钟；看到萌系界面，心情变好，愿意多玩几局
当前替代方案 原版界面视觉沉重（暗绿赌桌 + 橙色 CTA），对二次元用户缺乏亲切感；用户流失在第一印象

3核心价值主张

原版用"硬核牌局美学"留人；改造后用萌感 + 社区感（弹幕、吉祥物、樱花）留人。 对二次元玩家来说，界面本身就是内容，他们愿意为"好看好玩"多刷一局。

4功能清单（MVP）

所有功能均在单文件 index.html 内实现，不引入外部图片资源，不改动游戏逻辑。

作为玩家，当我打出同花顺/四条/葫芦/同花/顺子等强力手牌时，我会看到 3–5 条弹幕文字从屏幕右侧飘向左侧，模拟 B 站视频弹幕效果，让出大牌的瞬间更有仪式感。

  • 触发条件：皇家同花顺、同花顺、四条、葫芦、同花、顺子 中任意一种
  • 每次随机从弹幕文案池（见第 6 章）中抽 3–5 条
  • 动画：从 right: -200px 到 left: -200px，持续 3.5s，linear；每条随机 y 轴位置（避开 HUD 和 Action Bar）
  • 字体：白色字体 + 1px 黑色描边（text-shadow 模拟），字号 15px，Noto Sans SC
  • 多条弹幕之间有 200–500ms 随机延迟，避免全部同时出现
  • 弹幕层 z-index 高于牌面但低于 Modal/Joker 覆盖层

P0US-06 胜利樱花粒子

作为玩家，当我通关全部 3 关后，我会看到屏幕上撒下樱花粒子，粒子飘落过程约 2.5 秒，让胜利时刻更有庆祝感。

  • 粒子数量：20–30 个
  • 形态：用 SVG path 或 Unicode 字符 🌸 实现，每个粒子随机大小（12–24px）、随机初始 x 位置
  • 动画：从顶部自由落下 + 微小左右摆动（keyframe sin 曲线模拟），持续 2–2.5s
  • 粒子颜色：樱花粉 #FFB7C5、白色 #FFFFFF、浅粉 #FFDDEA 三色随机
  • 胜利 Modal 弹出时同步触发，粒子层在 Modal 背后（z-index 介于游戏层和 Modal 之间）

P0US-07 Web Audio 音效

作为玩家，当我执行出牌、选中牌、弃牌、胜利等操作时，我会听到轻快的提示音效，增强操作反馈感。全部通过 AudioContext + OscillatorNode + GainNode 合成，不依赖外部音频文件。

5像素科技背景需求 V2.1 NEW

5.1 风格定调：选用方案 E（C + B 混合）

在五个候选方案中，方案 E（Synthwave×16-bit 混合）与现有 V2.0 的 B站粉蓝主调、萌系二次元气质契合度最高——远景用 Synthwave 透视网格制造科技纵深感，近景保留 16-bit 像素颗粒与像素化樱花，两者叠加后既有"赛博未来"的酷感，又不丢失二次元的温度。

方案风格关键词代表作与 V2.0 契合度
A8-bit 复古 FC 风马里奥、FC 坦克低——色彩太少，萌感不足
B16-bit 现代像素Celeste、Hyper Light Drifter中——颜色丰富，但缺少科技感
CSynthwave 霓虹网格OutRun、Hotline Miami高——紫粉霓虹与 B站粉高度吻合
D赛博朋克字符雨黑客帝国、《2077》UI中——绿色调与 B站粉冲突
E ✓C×B 混合：远景 Synthwave + 近景 16-bit 像素—最高——两层分工明确，互不干扰

5.2 背景层次结构（由远至近）

Layer 0 · 底色渐变（保留自 V2.0）
深空底色 #0d0520 → #0a1628，从紫黑过渡到深海蓝，作为全局最底层。不做动画，纯 CSS 渐变。
z-index: 0position: fixedCSS linear-gradient
Layer 1 · 远景：Synthwave 透视网格
经典 Synthwave 透视地平线网格，从画面底部向上汇聚，营造无限延伸的科技纵深感。网格线颜色为紫粉霓虹 #bf5fff（约 20% opacity），线宽 1px，用 Canvas 2D 或纯 CSS repeating-linear-gradient 绘制。
动画：整体向玩家缓慢推进，周期约 8s，loop，产生行驶感。
z-index: 1Canvas 2D 或 CSS性能轻量
Layer 2 · 中景：像素化粒子（星点 + 像素樱花）
由两类元素构成：① 像素星点：20–30 个 2×2px 的正方形 div 随机分布，以随机周期（2–4s）在 0.2→1→0.2 opacity 之间闪烁，颜色为白色/青色/粉色随机。② 像素化樱花：将 V2.0 的 emoji 🌸 替换为 16×16px 的内联 SVG 像素花瓣（4色块拼合），从顶部缓慢飘落，左右微摆，持续循环。数量控制在 8–12 片，避免过密。
z-index: 2CSS animation内联 SVG
Layer 3 · 近景：CRT 扫描线 + 像素噪点膜
覆盖在背景最上层（UI 层之下）的半透明纹理层：① 扫描线：用 CSS repeating-linear-gradient 生成 2px 周期的水平条纹，opacity ≤ 8%，模拟 CRT 显示器效果，增加像素感而不抢眼。② 像素噪点：用 SVG feTurbulence filter 或 CSS 背景生成静态颗粒感，opacity ≤ 5%，仅在背景层可见，不叠加到卡牌/HUD。
z-index: 3CSS onlypointer-events: none
Layer 4+ · UI 层（chibi / 弹幕 / HUD / 卡牌）
所有 V2.0 已有 UI 元素保持原有 z-index 层级不变，背景层只在其下方工作。背景层的最大 z-index 不超过 4，UI 层从 10 起始。弹幕层（z-index: 50）和 Modal（z-index: 100+）维持现状。
z-index: 10+（现有 UI 不变）

5.3 现代科技感的具体表现

效果实现方案强度控制是否采用
霓虹辉光（Neon Glow） CSS box-shadow / text-shadow 多层叠加（模糊半径 4px/8px/16px，颜色 #bf5fff / #ff2d78） 仅用于透视网格边缘线 + 像素樱花高光，不用于卡牌/HUD 采用
网格透视（Perspective Grid） CSS perspective + rotateX 变换，或 Canvas drawLine 绘制汇聚线，配合 CSS animation 缓慢推进 整体 opacity ≤ 25%，不影响前景可读性 采用（Layer 1 核心实现）
CRT 扫描线（Scanlines） CSS repeating-linear-gradient(transparent, transparent 1px, rgba(0,0,0,0.08) 1px, rgba(0,0,0,0.08) 2px) opacity ≤ 8%，全局覆盖 采用（Layer 3）
像素闪烁动画 CSS @keyframes 控制 opacity 跳变（非渐变，产生像素 blink 感），用于星点粒子 闪烁频率 0.5–2Hz，不超过 3Hz（避免视觉疲劳） 采用（Layer 2 星点）
全息光斑（Holographic Flare） CSS radial-gradient 叠加，伪随机位置，颜色为青蓝/紫粉渐变 仅在胜利/大牌触发时短暂出现（0.5s），opacity ≤ 15% 采用（条件触发，非常驻）
故障艺术（Glitch） CSS transform: translateX 微小随机偏移 + 色差（R/B 通道偏移模拟） 仅用于大牌命中瞬间的短暂特效（≤0.3s），不常驻 可选（P2 评估，不纳入 MVP）
WebGL / 3D 渲染 — — 不采用（性能考虑，见第 11 章）

5.5 用户故事（背景层专项）

P0US-B1 远景透视网格沉浸感

作为玩家，当我打开游戏时，我会在背景最深处看到一张向地平线延伸的霓虹紫粉色透视网格，它缓慢向我推进，让我感受到"置身于某个赛博世界边缘"的科技氛围，而不是普通的棋牌室。

  • 实现方式：CSS perspective grid 或 Canvas 2D，8s 周期 loop
  • 颜色：#bf5fff 20% opacity，整体不喧宾夺主
  • 在 1080×640 游戏窗口内，网格地平线在垂直中点约 40% 处

P0US-B2 像素星点闪烁氛围

作为玩家，在游戏过程中，我会在背景中景区域看到稀疏的 2×2px 方形像素星点以随机频率闪烁，它们让整个画面有"活的"感觉，像是深空中的星星在呼吸，但不分散我对牌局的注意力。

  • 数量：20–30 个，随机分布，避开 HUD 和操作区
  • 颜色随机：白 / 青 / 粉三色，形态为 2×2px div（image-rendering: pixelated）
  • 闪烁动画：opacity 在 0.2–1 之间跳变（非渐变），频率 0.5–1.5Hz

P0US-B3 扫描线增强像素质感

作为玩家，当我仔细看游戏背景时，我能隐约感受到一层细密的水平扫描线纹理，像是在 CRT 显示器上游玩一款复古游戏，整个画面多了一丝"像素时代滤镜"，既复古又有现代科技感的反差萌。

  • 实现：CSS repeating-linear-gradient，2px 条纹周期，opacity ≤ 8%
  • 覆盖全屏背景层，但不覆盖卡牌/HUD/chibi 等 UI 元素
  • 静态，不做动画（滚动扫描线对性能影响大，且视觉易疲劳）

P0US-B4 像素化樱花与背景融合

作为玩家，当我看到背景中飘落的樱花时，我会看到它们以像素风格渲染（方块拼合的 SVG 花瓣），而不是圆润的 emoji，这与赛博像素风格的背景无缝融合，让游戏的整体美术风格统一。

  • 像素花瓣：16×16px 内联 SVG，用 4–6 个像素方块拼合花瓣形态
  • 颜色：樱花粉 #FFB7C5 + 高光白 #fff
  • 飘落动画复用 V2.0 逻辑，只替换 DOM 内容，数量 8–12 片（背景常驻版，轻于胜利版的 20–30 片）

P1US-B5 大牌触发全息光斑

作为玩家，当我打出同花顺/四条等强力手牌时，除了弹幕飘字之外，我会在背景层看到一个短暂的青蓝色全息光斑向四周扩散（约 0.5s），让出大牌的瞬间有一次视觉冲击波，增强仪式感。

  • 实现：CSS radial-gradient 配合 scale(0)→scale(1.5) + opacity(0.15→0) 动画
  • 颜色：rgba(0,255,255,0.15) 到透明
  • 触发时机与弹幕同步，持续 ≤ 0.5s，不阻塞操作

11不做什么

排除项原因
真人立绘 / 图片角色不引入外部图片资源，技术约束
BGM 背景音乐Web Audio 合成循环音乐复杂度高，与本次改造 ROI 不符；只做音效
移动端适配游戏固定 1080×640，移动端体验待二期评估
卡牌翻转动画需要重构发牌逻辑，影响游戏流稳定性，列二期
游戏规则/算分修改本次纯视觉/体验改造，不碰核心逻辑
登录/排行榜/存档无后端，纯静态，不在范围内
多语言当前仅服务简中用户
无障碍（a11y）优化本次不系统处理，仅保证不破坏原有键盘操作
WebGL / Three.js 渲染 V2.1 新增引入 WebGL 会带来额外依赖，增加首屏加载时间，且超出"纯静态无构建"约束；所有背景效果必须用 CSS / Canvas 2D / SVG 实现
全屏 Canvas 粒子系统 V2.1 新增大量 Canvas 粒子（>100 个）会在低端设备上造成明显掉帧；背景粒子总数控制在 30 个以内，优先 CSS animation
外部图片/纹理文件引入 V2.1 新增所有像素风元素必须通过内联 SVG 或纯 CSS 实现，不允许 <img src> 或 url() 引用外部文件，确保单文件纯静态部署
常驻 Glitch 故障艺术动画 V2.1 新增Glitch 特效若常驻会严重干扰游戏可读性，仅允许作为大牌触发的瞬时特效（≤0.3s），MVP 阶段列为 P2 可选
滚动扫描线（动态） V2.1 新增动态滚动扫描线对 GPU 的合成层开销较大，且长时间观看易造成视觉疲劳；扫描线必须为静态 CSS 纹理

13开放问题

#问题需决策方影响范围
Q1 chibi 的发型/装扮风格（马尾？双马尾？猫耳？）应在 SVG 设计前确认，影响整体气质定调 产品/视觉 US-04
Q2 弹幕是否需要"过滤机制"——同一局重复出大牌时，弹幕是否会重复？是否需要去重？ 产品 US-05
Q3 音效音量是否需要提供设置入口（静音按钮）？部分用户可能在公共场所玩 产品 US-07
Q4 全通关胜利时，Modal 弹出和樱花粒子之间的时序：樱花先飘还是 Modal 先出，需要对齐 产品/开发 US-06
Q5 chibi 对话中部分文案涉及日文（例如「すごい」）是否在吉祥物对话中也使用中日混合，还是仅弹幕使用？ 产品 US-04
Q6 透视网格的实现方式：CSS perspective + rotateX（纯 CSS，无 JS 依赖）还是 Canvas 2D drawLine（更精细，可控）？CSS 方案在高 DPR 屏上可能出现锯齿，需测试后定夺 开发 US-B1
Q7 背景层动画与游戏主循环的线程隔离：是否需要把背景动画独立放入 Web Worker？目前预估不需要，但高帧率设备上 will-change: transform 提示是否足够，待性能测试后确认 开发 A13
